# Practical Workspace Template


## Reader Outcome

After this section, the reader can:

- Створити мінімальний файловий workspace, яким може навігувати агент.
- Відокремити універсальні стартові правила від специфічних для проєкту правил.
- Визначити профіль можливостей проєкту для доступності Git, компілятора, тестів, runtime та статичних перевірок.
- Використовувати файли `idea.md` проєкту та модуля як одноразові bootstrap-входи.
- Писати кореневі та модульні файли інструкцій, не вигадуючи непідтримуваний процес.
- Реєструвати модулі, корені wiki, корені джерел і корені артефактів.
- Додавати шаблони, потрібні для перших концепцій модуля, контрактів таблиць, контрактів класів, сценаріїв використання, дизайнів, планів і нотаток сесії.
- Додавати мітки впевненості до важливих тверджень wiki та контрактів.

## Core Explanation

Практична стартова точка — це не теорія шарів документації. Це workspace, який каже агенту, з чого почати, що читати, чого уникати надлишкового читання, де належить довготривале знання і що має бути перевірено, перш ніж роботу назвуть завершеною. Workspace — це детермінована система маршрутизації контексту, а не архів документації.

Стартовий шаблон слід копіювати з реальних робочих артефактів, коли це можливо. Універсальні частини можна повторно використовувати майже без змін. Специфічні для проєкту частини слід чітко позначати як приклади, а не правила для кожної команди. У цьому проєкті перевірки кодування, відсутність git, відсутня верифікація компілятора/runtime, папки експорту таблиць і псевдоніми модулів є специфічними для проєкту. Патерн наявності цих правил універсальний; точні правила — ні.

Корисний стартовий workspace має чотири завдання:

- орієнтація: визначення активного модуля та локальних правил;
- пам'ять: збереження компактного контексту перезапуску та довготривалого знання;
- свідчення: підтримка сирих і оброблених вихідних артефактів у стані, придатному для аудиту;
- впевненість: позначення важливих тверджень як confirmed, inferred, runtime-unverified чи blocked-by-access;
- інтеграція робочого процесу: надання Superpowers чи еквівалентному робочому процесу достатнього проєктного контексту для планування, реалізації, верифікації та оновлення документації.

## Starter Structure

Використовуйте це як мінімальну форму, потім адаптуйте шляхи до проєкту:

```text
workspace/
  AGENTS.md
  idea.md        # bootstrap input, not recurring memory
  known_issues.md
  modules.md
  session_state.md
  docs/
    superpowers/
      specs/
      plans/
  knowledge-base/
    INDEX.md
    shared/
    modules/
      <module>/
  source-artifacts/
    INDEX.md
    tables/
      raw/
      processed/
  <module>/
    AGENTS.md
    idea.md      # bootstrap input, not recurring memory
    session_state.md
    docs/
```

Для модуля методології, що є лише документацією, цей проєкт також використовує:

```text
Methodology/
  AGENTS.md
  Methodology/
    sections/
  Templates/
  Wiki/
  Assembly/
  Output/
  session_state.md
```

Ця друга форма специфічна для проєкту. Вона корисна, коли сама документація є продуктом і повинна збиратися з вихідних розділів.

## Root Instructions Template

Кореневий `AGENTS.md` повинен спершу визначати універсальну поведінку:

```markdown
# Workspace Rules

- Read `known_issues.md` before non-trivial work.
- Read `modules.md` to resolve module names, aliases, paths, wiki roots, source roots, and artifact roots.
- If the active module is unclear, ask which module to use.
- If no module is explicitly requested, continue with the last active module named in root `session_state.md`.
- Read the active module `AGENTS.md` before non-trivial work.
- Read only the latest entry in the active module `session_state.md`: stop at `<!-- AGENT-SESSION-STOP -->`.
- Keep module-specific durable knowledge in the module wiki.
- Keep reusable platform or cross-module knowledge in shared knowledge.
- Keep raw source artifacts separate from curated knowledge.
- Update the active module `session_state.md` after every non-conversational task.
- Update root `session_state.md` only as a compact module activity index.
- When full automated testing is impossible, document static checks and remaining runtime assumptions.
- Keep `session_state.md` even when Git exists; Git records file history, while session state records task intent, evidence, assumptions, verification, and next action.
```

Кореневий `idea.md` не є частиною регулярного шляху читання. Використовуйте його як одноразовий bootstrap-вхід: користувач може сказати: "Ось ідея проєкту; прочитай її, постав питання, якщо щось неясно, і створи потрібні стартові артефакти." Після bootstrap довготривале значення повинно жити в `modules.md`, модульних інструкціях, сторінках концепції модуля, планах і нотатках сесії.

Кореневий `session_state.md` повинен залишатися компактним. Його завдання — сказати, який модуль був активний найостанніше, і індексувати нещодавню роботу на рівні модуля, а не зберігати кожну деталь. Хороший запис називає модуль, довготривалі сторінки чи артефакти, яких торкнулася робота, виконану верифікацію та наступну корисну дію. Довготривалі факти повинні переходити в модульне чи спільне знання, а `session_state.md` посилатиметься на них.

Потім додайте специфічні для проєкту правила в тому самому файлі, чітко позначені:

```markdown
## Project-Specific Rules

- Capability profile:
  - Git: unavailable. Use filesystem/static verification instead of `git status` or `git diff`.
  - Compiler/build: unavailable. Use syntax-aware static scans where possible and record remaining assumptions.
  - Automated tests: unavailable or incomplete. Name any manual or static checks used.
  - Runtime access: unavailable locally. Mark behavior that depends on target runtime settings as runtime-unverified.
- Example: Preserve UTF-8 for Cyrillic text and run the project's encoding scan after Markdown/source edits.
- Example: Use the project's table artifact folders for XML/CSV/TXT exports.
- Example: Use the project's wiki curator skill or script when normalizing source evidence.
```

Наведені вище приклади походять із цього проєкту та являють собою найважчий базовий випадок. Інший проєкт може замінити їх правилами діалекту SQL, обмеженнями системи збірки, обмеженнями розгортання, правилами конфіденційності даних чи правилами доступу до runtime. Якщо Git існує, визначте, як використовувати локальний статус, diff'и та нещодавню історію. Якщо існує компілятор, команда збірки, test runner чи локальний runtime, визначте точні команди і зробіть їх частиною правила верифікації. Не видаляйте `session_state.md`: історія Git, логи збірки та вивід тестів не надійно зберігають, чому завдання відбулося, які свідчення використовувалися, які припущення залишаються чи що повинно статися далі.

## Module Instructions Template

Кожен модуль повинен мати менший `AGENTS.md`:

```markdown
# <Module> Module Rules

- Read `../AGENTS.md` first.
- Read `../known_issues.md` before non-trivial work.
- Read `../modules.md` to resolve module paths.
- Module: <Module>.
- Module wiki lives in `<module-wiki-root>/`.
- Supporting docs, specs, and plans live in `<module>/docs/`.
- Work on one task or section at a time.
- Update `session_state.md` after every non-conversational module task.
- Preserve project encoding and verification rules.
```

Потім додайте специфічні для модуля правила:

```markdown
## Module-Specific Rules

- Example: This module is documentation-only.
- Example: Source sections live in `Methodology/sections/`.
- Example: Final assembled output must not be edited directly except during mechanical assembly.
- Example: Avoid project-sensitive source-origin wording; use neutral evidence names.
```

Знову ж таки, ці приклади з цього проєкту. Універсальна частина — це структура: модульні правила успадковують кореневі правила, визначають модуль, називають локальні корені знань і фіксують специфічні для модуля обмеження.

Модульний `idea.md` має ту саму одноразову роль, що й ідея проєкту. Використовуйте його, коли користувач просить агента здійснити bootstrap модуля з файлу ідеї чи тексту ідеї. Після того як концепція модуля та початкові артефакти маршрутизації існують, агенти зазвичай повинні читати wiki модуля та стан сесії замість повторного читання оригінальної ідеї.

Модульний `session_state.md` детальніший за кореневий, але це все ще коротка пам'ять. Він повинен містити лише останній контекст перезапуску перед стоп-маркером: поточне завдання чи розділ, використаний вихідний матеріал, рішення, відкриті питання, верифікацію та наступну дію. Старіша історія може залишатися нижче маркера для цілей аудиту, але агенти не повинні розглядати її як поточну інструкцію.

## Module Registry Template

`modules.md` повинен бути коротким і механічним:

```markdown
# Modules

- <ModuleName>
  - path: `<module-path>/`
  - wiki: `<module-wiki-root>/`
  - shared wiki: `<shared-wiki-root>/`
  - sources: `<source-root>/`, `<example-source-root>/`
  - artifacts: `<artifact-root>/`
  - aliases: <natural language aliases>
```

Це файл, що перетворює природну мову на шляхи. Тримайте псевдоніми практичними: назви, які користувачі справді вводять, скорочення та доменні прізвиська.

## Required Templates

Мінімум створіть шаблони для:

- сторінки концепції модуля;
- сторінки таблиці;
- сторінки контракту класу чи функції;
- сторінки сценарію використання;
- документа дизайну;
- плану реалізації;
- запису стану сесії.

Шаблони wiki та контрактів повинні відокремлювати статус сторінки від впевненості твердження. `status` каже, чи сторінка достатньо повна для використання. `confidence` каже, наскільки конкретний факт підтверджений: `confirmed`, `inferred`, `runtime-unverified` чи `blocked-by-access`.

Файли `idea.md` проєкту та модуля є рекомендованими bootstrap-входами, особливо для нових workspace. Вони не є регулярними операційними артефактами, і зрілі модулі можуть не потребувати ретроактивних файлів ідеї, якщо їхній намір уже зафіксовано в довготривалих сторінках wiki та планах.

Цей проєкт зберігає копійовані шаблони project-workspace у `Methodology/Templates/project-workspace/templates/`. Інший проєкт може зберігати їх під кореневим `templates/` чи папкою docs модуля, але шаблони повинні бути легко доступними для агентів.

Цей проєкт також постачає копійований пакет project workspace у `Methodology/Templates/project-workspace/`. Він містить кореневі та модульні файли `idea.md`, кореневі стартові файли, один приклад модуля, корені знань, папки вихідних артефактів, папки робочого процесу та багаторазові шаблони сторінок, згруповані в папки шаблонів `wiki`, `workflow` і `session`. Розглядайте це як стартову точку для адаптації, а не як завершену конфігурацію проєкту.

Корінь `Methodology/Templates/` зарезервований для шаблонів, що використовуються для виробництва самої цієї методології, таких як `section-contract.md` і `section-dod.md`. Це специфічні для продукту шаблони для написання цієї книги з файлів розділів. Вони не потрібні кожному ERP-проєкту, якщо тільки цей проєкт також не виробляє зібрану документацію з вихідних розділів.

## Reproducible Procedure

1. Створіть кореневу структуру.
   - Додайте кореневі інструкції, кореневу ідею, відомі проблеми, реєстр модулів, кореневий стан сесії, docs, спільне знання, знання модуля, вихідні артефакти та принаймні одну папку модуля.

2. Напишіть ідею проєкту.
   - Зафіксуйте призначення проєкту, аудиторію, бізнес-обсяг, початкові модулі, наявні свідчення, очікування щодо робочого процесу, відкриті питання та перший корисний наступний крок.
   - Тримайте її достатньо короткою, щоб почати діалог, а не замінити пізніші дизайни чи плани.
   - Розглядайте її як вхід для bootstrap-розмови, а не як файл, який кожен агент повинен читати завжди.

3. Додайте кореневі інструкції.
   - Почніть з універсального шаблону.
   - Додайте чітко позначений специфічний для проєкту розділ.
   - Включіть профіль можливостей для Git, компілятора/збірки, автоматизованих тестів, доступу до runtime, статичних перевірок і правил runtime-припущень.

4. Додайте перший модуль.
   - Створіть ідею модуля, інструкції модуля та стан сесії модуля.
   - Зареєструйте модуль у `modules.md`.
   - Створіть корінь wiki модуля чи вкажіть на усталене місце wiki модуля.

5. Напишіть ідею модуля.
   - Зафіксуйте призначення модуля, користувачів і дійових осіб, бізнес-можливості, межі, основні терміни, типові стартові запити, перші свідчення, відкриті питання та наступну дію.
   - Використовуйте її як джерело для першої сторінки концепції модуля.
   - Для зрілих наявних модулів пропускайте ретроактивне створення файлу ідеї, коли той самий намір уже живе в курованому знанні модуля.

6. Додайте шаблони стану сесії.
   - Створіть компактний шаблон кореневого запису для останнього активного модуля, нещодавніх довготривалих змін, верифікації та наступної дії.
   - Створіть шаблон модульного запису для поточного завдання чи розділу, вихідного матеріалу, рішень, відкритих питань, верифікації та наступної дії.
   - Включіть правило стоп-маркера для історій модулів, що накопичують записи.

7. Додайте стартові шаблони.
   - Створіть шаблони концепції модуля, таблиці, контракту, сценарію використання, дизайну, плану та стану сесії.
   - Тримайте шаблони достатньо короткими для повторного використання, не відшліфовуйте їх до прозових есе.

8. Додайте папки життєвого циклу артефактів.
   - Створіть зони сирих і оброблених вихідних артефактів.
   - Створіть індекс артефактів.
   - Зазначте, коли сирі артефакти можуть переходити в оброблені.

9. Додайте специфічні для проєкту захисні правила.
   - Правила кодування, обмеження SQL/runtime, обмеження пропрієтарного доступу, обмеження збірки/тестів, правила конфіденційності, наявний інструментарій чи відсутній інструментарій належать сюди.
   - Позначайте їх як приклади при публікації загального гайду.

10. Проведіть аудит шаблонів перед тим, як назвати стартовий комплект завершеним.
   - Перевірте, що кожен шаблон, згаданий у гайді, існує.
   - Перевірте, що кожен поставлений шаблон пояснено в гайді.
   - Перевірте, що специфічні для проєкту правила не подані як універсальні.

## Required Artifacts

- Кореневий `AGENTS.md`.
- `known_issues.md`.
- `modules.md`.
- Кореневий `session_state.md`.
- Принаймні один модульний `AGENTS.md`.
- Принаймні один модульний `session_state.md`.
- Спільний і модульний корені знань.
- Корені сирих і оброблених вихідних артефактів плюс індекс артефактів.
- Стартові шаблони сторінок.
- Копійований стартовий пакет workspace при публікації методології.
- Скрипти верифікації чи захисних перевірок, коли проєкт їх вимагає.

Рекомендовані bootstrap-артефакти:

- Кореневий `idea.md` для нового проєкту.
- Модульний `idea.md` для нового модуля.

Опціональні артефакти:

- Шаблон сторінки патерну чи папка патернів, коли багаторазові патерни стають достатньо поширеними, щоб виправдати окрему сім'ю сторінок.

## Example

У цьому проєкті кореневі інструкції містять універсальні правила маршрутизації та специфічні для проєкту правила. Універсальні правила включають читання `modules.md`, визначення активного модуля, читання лише останнього запису стану сесії модуля, збереження стану сесії навіть якщо існує система контролю версій, і запис довготривалого знання в модульну чи спільну wiki. Специфічні для проєкту правила включають відсутність git, відсутність локальної верифікації компілятора/runtime, сканування кодування для чутливих до кирилиці змін і життєвий цикл артефактів таблиць.

Та сама ідея маршрутизації застосовується всередині сторінок. Концепція модуля повинна маршрутизувати від мови користувача до таблиць, контрактів і сценаріїв. Сторінка таблиці повинна маршрутизувати від бізнес-термінів до полів, свідчень і використання. Сторінка контракту повинна маршрутизувати від назв API до поведінки, побічних ефектів і прогалин верифікації. Сценарій використання повинен маршрутизувати від сценарію до уражених контрактів, таблиць, перевірок і залишкової невизначеності.

Модуль Methodology потім додає локальні правила: він лише документація, вихідні розділи живуть у `Methodology/sections/`, шаблони живуть у `Templates/`, кінцевий вихід живе у `Output/`, і вихід не слід редагувати безпосередньо, крім як під час механічної збірки.

Новий проєкт-адоптер почав би на один рівень раніше: кореневий `idea.md` заявляє, навіщо існує workspace і які перші модулі важливі; перший модульний `idea.md` заявляє бізнес-призначення модуля та стартові запити. Концепція модуля потім перетворює цей початковий намір на довготривалу пам'ять модуля з термінами, межами, шаблонами наміру, посиланнями та інваріантами. Наявні зрілі модулі можуть пропустити ретроактивне створення `idea.md`, коли їхній намір уже живе в курованому знанні.

Кореневий `session_state.md` каже, що Methodology був останнім активним модулем, і підсумовує нещодавню активність модуля. `Methodology/session_state.md` несе контекст перезапуску для поточної роботи з документацією: які розділи змінилися, які перевірки пройшли і яка наступна дія залишається. Жоден із файлів не є довготривалим домом для правил методології; ним є вихідні розділи, сторінки wiki та шаблони.

Цей поділ — патерн для копіювання: тримайте структуру маршрутизації універсальною і чесно позначайте обмеження проєкту.

## Failure Modes

- Публікація специфічних для проєкту правил так, ніби кожен проєкт повинен їх використовувати.
- Початок з великої wiki до того, як агент має інструкції маршрутизації.
- Написання сторінок, що пояснюють тему, але не маршрутизують до пов'язаних контрактів, артефактів, міток впевненості чи прогалин верифікації.
- Пропуск псевдонімів модуля, через що запити природною мовою не потрапляють на правильний шлях.
- Пропуск явного стартового наміру, через що перший діалог починається з файлів замість бізнес-призначення.
- Зберігання сирих експортів у wiki замість папок вихідних артефактів.
- Постачання шаблонів, які гайд ніколи не пояснює.
- Пояснення в гайді артефактів, яких немає в стартовому комплекті.
- Копіювання стартового workspace без заміни заповнювачів і специфічних для проєкту прикладів.
- Розгляд зібраного виходу як редагованого джерела.
- Дозвіл `session_state.md` стати прихованою wiki замість нотатки перезапуску та індексу активності.
- Розгляд історії Git, логів збірки чи виводу тестів як заміни стану сесії.
- Неувімкнення сильнішої верифікації, коли Git, компілятор/збірка, тести чи доступ до runtime доступні.

## Assumptions And Runtime Limits

- Цей шаблон передбачає файловий workspace, де агенти можуть читати та оновлювати файли Markdown.
- Назви папок можуть змінюватися, але відповідальності повинні залишатися явними.
- Деякі проєкти можуть потребувати сильніших правил контролю доступу, конфіденційності чи runtime, ніж показує цей стартовий комплект.
- Захисні перевірки кодування, недоступний git, недоступні перевірки компілятора/збірки, покриття тестами та обмеження доступу до runtime є специфічними для проєкту обмеженнями, а не універсальними припущеннями.
