# Workspace Bootstrap


## Reader Outcome

After this section, the reader can:

- Створити мінімальний набір файлів workspace, який робить модель пам'яті операційною.
- Пояснити призначення кожного bootstrap-файлу перед його створенням чи використанням.
- Реєструвати модулі зі шляхами, псевдонімами, коренями wiki, коренями джерел і коренями артефактів.
- Додавати маркери, індекси, псевдоніми та умовні інструкції читання, які запобігають надлишковому читанню.
- Створювати файли інструкцій агента останніми, після того як файли, на які вони посилаються, вже визначені.

## Core Explanation

Система знань визначає шари пам'яті. Workspace bootstrap перетворює ці шари на конкретні файли та папки.

Bootstrap не повинен починатися з файлу правил, який посилається на непояснені файли. Спочатку визначте артефакти, які координуватиме файл правил. Потім створіть файл інструкцій агента, який каже агенту, як їх використовувати. Це робить налаштування зрозумілим і слідує локальному правилу: визначай кожен об'єкт перед використанням у інструкціях.

Основні bootstrap-об'єкти:

- `known_issues.md`: короткий операційний список повторюваних ризиків workspace та необхідної обробки.
- `modules.md`: реєстр модулів; він відображає назви модулів і псевдоніми на шляхи, корені wiki, корені джерел, спільні корені wiki та корені артефактів.
- `session_state.md`: short memory; зберігає компактний контекст перезапуску і зупиняється на `<!-- AGENT-SESSION-STOP -->`.
- Профіль можливостей проєкту: короткий запис доступних інструментів верифікації та історії, таких як Git, компілятор, автоматизовані тести, доступ до runtime, знімки бази даних, перевірки кодування та експортери артефактів.
- Корінь wiki модуля: module memory для локальних концепцій, контрактів, сценаріїв використання, рішень і відкритих питань.
- Спільний корінь wiki: shared product memory для багаторазових знань платформи чи між модулями.
- Папки вихідних артефактів: області source evidence для сирих і оброблених експортів.
- Папки дизайну та планів: medium memory для міркувань завдання.
- Файл інструкцій агента: локальні правила для агента. У цьому workspace це файл `AGENTS.md`; в іншому середовищі агента це може бути `CLAUDE.md`, `GEMINI.md`, інструкції workspace чи інший підтримуваний файл правил.

Після того як ці об'єкти існують, файл інструкцій агента може безпечно на них посилатися. Він стає контрактом орієнтації: прочитати відомі проблеми, визначити модуль, завантажити лише останній стан сесії, використовувати правильні корені пам'яті, зберегти кодування і зафіксувати обмеження верифікації.

Найбезпечніше значення за замовчуванням — найважчий випадок: припускайте, що може не бути історії Git, компілятора, локального runtime і повного набору автоматизованих тестів, доки проєкт не доведе протилежне. Це значення за замовчуванням тримає метод придатним для використання в обмежених пропрієтарних середовищах. Це не привід ігнорувати кращі інструменти, коли вони існують. Якщо Git доступний, використовуйте його для історії змін файлів і diff'ів. Якщо доступні компілятор, команда збірки, test runner чи цільовий runtime, зробіть ці перевірки частиною правил верифікації. Профіль можливостей фіксує, які з цих інструментів доступні і де агент має їх використовувати.

`session_state.md` залишається обов'язковим навіть у проєкті з увімкненим Git. Історія Git відповідає на питання контролю версій: які файли змінилися, хто закомітив, коли стався коміт і який diff зафіксовано. Short memory відповідає на питання безперервності завдання, на які коміти часто не відповідають: чому робота була розпочата, які свідчення були використані, які припущення залишаються неперевіреними, що обговорювалося без зміни коду, які перевірки були пропущені і яка наступна корисна дія. Не кожне дослідження стає комітом, не кожне корисне відкриття змінює файли, і повідомлення комітів не призначені як контекст перезапуску агента.

Bootstrap також реалізує шаблон маршрутизації контексту з розділу про систему знань. Файлів недостатньо. Workspace має включати стоп-маркери для файлів історії, що зростають, індекси для коренів wiki, псевдоніми для назв модулів природною мовою, реєстри артефактів для вихідних свідчень та інструкції, які кажуть, коли читати опціональний чи дорогий контекст. Ці засоби контролю не дають "доступному контексту" перетворитися на "обов'язковий контекст".

## Reproducible Procedure

1. Створіть `known_issues.md`.
   - Мета: повторювані ризики, які кожен майбутній агент повинен обробляти.
   - Тримайте його коротким і операційним.
   - Фіксуйте лише повторювані проблеми workspace, які майбутні агенти повинні обробляти.
   - Для кожної проблеми назвіть симптом, причину та необхідну обробку.
   - За потреби посилайтеся на детальне обґрунтування у спільних wiki-сторінках.

2. Створіть `modules.md`.
   - Мета: таблиця маршрутизації від назв модулів природною мовою до шляхів workspace.
   - Зареєструйте кожен модуль під канонічною назвою.
   - Для кожного модуля зафіксуйте шлях модуля, корінь wiki, спільний корінь wiki, корені джерел, корені артефактів і псевдоніми.
   - Використовуйте псевдоніми, які користувачі природно вводять, включно з назвами продуктів, короткими назвами та доменними мітками.

3. Створіть кореневий `session_state.md`.
   - Мета: компактний контекст перезапуску workspace.
   - Розмістіть поточний запис першим.
   - Включіть `Last active module`.
   - Завершуйте кожен запис точним маркером `<!-- AGENT-SESSION-STOP -->`.
   - При майбутніх читаннях зупиняйтеся на першому маркері, щоб старий контекст не перевизначав поточний стан.

4. Створіть профіль можливостей проєкту.
   - Мета: зробити доступні інструменти явними, а не назавжди припускати найважче середовище.
   - Почніть із базового найважчого випадку: немає Git, немає компілятора, немає локального runtime, немає надійних автоматизованих тестів.
   - Якщо Git існує, визначте, коли використовувати `git status`, `git diff` та перегляд останнього логу.
   - Якщо існує компілятор, команда збірки, test runner чи runtime, визначте точні команди верифікації і що означає їхній успішний чи неуспішний результат.
   - Якщо інструмент існує лише для деяких модулів, зафіксуйте це на рівні модуля, а не як правило для всього workspace.
   - Тримайте `session_state.md` обов'язковим, бо він фіксує намір завдання, свідчення, припущення, верифікацію і наступну дію, а не лише зміни файлів.

5. Створіть корені модулів.
   - Мета: обмежені робочі області з власною пам'яттю та правилами.
   - Створіть кожну папку модуля.
   - Створіть `session_state.md` кожного модуля для короткої пам'яті модуля.
   - Тримайте запис стану модуля зосередженим на поточному статусі, використаному вихідному матеріалі, рішеннях, відкритих питаннях, верифікації та наступній дії.

6. Створіть корені wiki.
   - Мета: long memory.
   - Використовуйте `knowledge-base/shared` для багаторазових знань платформи чи між модулями.
   - Використовуйте `knowledge-base/modules/<module>` для специфічних для модуля знань, якщо модуль не має явного локального кореня wiki.
   - Додайте файли `INDEX.md`, щоб агенти могли дешево входити в wiki.
   - Додайте посилання пов'язаних нотаток від сторінок-резюме до глибших сторінок свідчень.

7. Створіть папки вихідних артефактів.
   - Мета: source evidence.
   - Розміщуйте щойно знайдені експорти XML, CSV, TXT чи подібні в `source-artifacts/tables/raw`.
   - Переносьте розпізнані експорти в `source-artifacts/tables/processed` лише після того, як wiki-сторінки створено чи оновлено.
   - Підтримуйте `source-artifacts/INDEX.md` як реєстр артефактів.

8. Створіть папки дизайну та планів.
   - Мета: medium memory для специфічних для завдання міркувань.
   - Використовуйте кореневий `docs/` для дизайнів і планів усього workspace.
   - Використовуйте модульний `docs/`, коли документ належить одному модулю.

9. Створіть кореневий файл інструкцій агента.
   - Мета: сказати агенту, як орієнтуватися в workspace.
   - У цьому workspace використовуйте кореневий `AGENTS.md`.
   - В інших середовищах використовуйте механізм файлу правил, підтримуваний цим агентом, наприклад `CLAUDE.md`, `GEMINI.md`, інструкції workspace чи еквівалент.
   - Визначте порядок орієнтації: прочитати відомі проблеми, визначити активний модуль, прочитати правила модуля, завантажити останній стан сесії і використовувати правильні корені пам'яті.
   - Визначте умовні правила читання для коренів джерел, коренів wiki, історичних нотаток та опціональних супровідних документів.
   - Визначте очікування верифікації для профілю можливостей проєкту, включно з тим, що робити, коли Git, компілятор, тести чи пропрієтарний runtime недоступні.
   - Додайте обробку кодування для файлів, чутливих до кирилиці.

10. Створіть файл інструкцій агента для кожного модуля.
   - Мета: локальні операційні правила модуля.
   - У цьому workspace використовуйте `<module>/AGENTS.md`.
   - Явно назвіть модуль.
   - Визначте специфічні для модуля папки продукту, папки wiki, папки артефактів і папки docs.
   - Вкажіть, чи модуль створює код, є лише документацією, чи лише курує артефакти.
   - Вимагайте оновлення `session_state.md` модуля після неконверсаційної роботи.

11. Додайте статичні захисні перевірки верифікації.
   - Мета: компенсувати обмежений доступ до інструментів чи runtime.
   - Зберігайте UTF-8 для активних файлів Markdown і джерел, якщо робочий процес з артефактами не вимагає іншого кодування.
   - Додайте чи задокументуйте перевірку кодування для змін, чутливих до кирилиці.
   - Якщо Git, перевірки компілятора, автоматизовані тести чи повні runtime-тести неможливі, вимагайте статичних перевірок і явних припущень.

## Required Artifacts

- Кореневий `known_issues.md`.
- Кореневий `modules.md`.
- Кореневий `session_state.md`.
- Профіль можливостей проєкту чи специфічні для проєкту правила доступності інструментів.
- Модульний `<module>/session_state.md`.
- Спільний корінь wiki, зазвичай `knowledge-base/shared`.
- Корінь wiki модуля, зазвичай `knowledge-base/modules/<module>` чи специфічний для модуля шлях wiki.
- Папки сирих і оброблених артефактів, зазвичай `source-artifacts/tables/raw` і `source-artifacts/tables/processed`.
- Індекс артефактів, зазвичай `source-artifacts/INDEX.md`.
- Папки дизайну та планів.
- Сторінки `INDEX.md` wiki та посилання пов'язаних нотаток.
- Стоп-маркер стану сесії, точно `<!-- AGENT-SESSION-STOP -->`.
- Умовні правила читання для опціонального чи дорогого контексту.
- Кореневий файл інструкцій агента, наприклад `AGENTS.md`, `CLAUDE.md`, `GEMINI.md` чи еквівалент.
- Файл інструкцій агента модуля, наприклад `<module>/AGENTS.md` чи еквівалент.
- Робочий процес кодування/статичної верифікації, наприклад `tools/check_encoding.ps1`.
- Команди верифікації збірки, компілятора, тестів, runtime чи Git, коли цільовий проєкт їх має.

## Example

Приклад пропрієтарного ERP-workspace використовує таку форму bootstrap:

```text
Workspace/
  known_issues.md
  modules.md
  session_state.md
  AGENTS.md
  knowledge-base/
    shared/
    modules/
      Alpha/
      Beta/
  source-artifacts/
    tables/
      raw/
      processed/
  Methodology/
    AGENTS.md
    session_state.md
    Wiki/
    Methodology/sections/
```

`modules.md` реєструє `Alpha`, `Beta` і `Methodology`. Якщо користувач каже "продовжити роботу над методологією" і шлях модуля не надано, кореневий `session_state.md` визначає останній активний модуль як `Methodology`. Модульний `AGENTS.md` тоді каже, що це модуль лише документації, що вихідні розділи живуть у `Methodology/Methodology/sections/`, і що робота повинна вестися по одному розділу за раз.

Це невеликий приклад, але він відображає важливу поведінку: агент не робить висновок про модуль лише з відкритих вкладок редактора, не читає весь історичний лог сесій і випадково не кладе специфічні для модуля факти у спільні знання.

Він також відображає поведінку проти надлишкового читання. `modules.md` маршрутизує псевдоніми природної мови до одного модуля. `session_state.md` зупиняється на поточному записі. Сторінки `INDEX.md` маршрутизують агента в куровані wiki-сторінки. Індекси артефактів маршрутизують від назви таблиці до розпізнаних свідчень. Інструкції агента кажуть, коли читати глибші джерела, а коли залишатися з курованою пам'яттю.

Файл інструкцій агента з'являється після того, як визначено `known_issues.md`, `modules.md`, `session_state.md`, корені wiki та корені артефактів. Цей порядок навмисний: файл правил координує відомі об'єкти замість того, щоб представляти їх несподівано.

У проєкті з Git і компілятором той самий bootstrap додав би специфічний для проєкту профіль можливостей. Кореневі правила казали б, які команди Git можна використовувати для перегляду локальних змін, яка команда збірки чи компілятора перевіряє синтаксис, які тести надійні і які runtime-перевірки все ще потребують людини чи цільового середовища. Правило стану сесії залишилося б на місці, бо агенту все ще потрібен контекст перезапуску для досліджень, обговорень дизайну, припущень і наступних дій, які можуть ніколи не з'явитися в коміті.

## Failure Modes

- Створення інструкцій агента першими: файл посилається на об'єкти, яких читач ще не розуміє.
- Відсутність файлу відомих проблем: повторювані ризики повторно відкриваються вже після того, як шкоду завдано.
- Відсутність реєстру модулів: запити природною мовою не можна надійно маршрутизувати до шляхів, джерел, коренів wiki чи коренів артефактів.
- Відсутність псевдонімів: користувачі вводять доменні назви, очевидні для команди, але невидимі для агента.
- Відсутність стоп-маркера в стані сесії: старі рішення та застарілі відкриті питання завантажуються так, ніби вони поточні.
- Стан сесії стає щоденником: short memory стає дорогою і шумною.
- Лог Git розглядається як пам'ять сесії: від історії контролю версій вимагають пояснити намір завдання, відкриті припущення чи незакомічені дослідження, яких вона надійно не містить.
- Доступні інструменти не зафіксовані: агент продовжує використовувати статичні перевірки для найважчого випадку, навіть коли існує компілятор, test runner, історія Git чи runtime-перевірка.
- Відсутність індексів чи пов'язаних посилань: папки wiki стають завданнями для широкого читання.
- Відсутність умовних правил читання: корені джерел та історичні нотатки стають контекстом за замовчуванням замість опціональних свідчень.
- Спільні та модульні знання змішані: багаторазові платформні факти застряють в одному модулі, або специфічні для модуля правила надмірно узагальнюються.
- Сирі артефакти редагуються на місці: вихідні свідчення стає важко аудіювати після розпізнавання.
- Відсутні захисні перевірки кодування: текст кирилицею може перетворитися на "кракозябри" під час, здавалося б, непов'язаних правок документації чи джерел.
- Runtime-обмеження приховані: статичні перевірки помилково сприймаються як повна runtime-верифікація.

## Assumptions And Runtime Limits

- Цей метод передбачає файловий workspace, де агенти можуть читати та оновлювати правила Markdown, wiki-нотатки та вихідні артефакти.
- Git корисний, але не обов'язковий. У workspace найважчого випадку git не очікується, тож верифікація покладається на перевірки файлової системи, статичні сканування та нотатки стану сесії.
- Доступ до компілятора, збірки, тестів і runtime корисний, але не гарантований. Коли вони існують, інструкції проєкту повинні називати точні команди і вимагати від агента їх використання там, де це релевантно.
- Пропрієтарна runtime-поведінка може залишатися неперевіреною. Коли це трапляється, обмеження має бути назване в розділі, плані, wiki-сторінці чи стані сесії, які від неї залежать.
- Bootstrap сам по собі не документує бізнес-поведінку. Він лише створює довготривалі місця, де бізнес-поведінку можна задокументувати.
