# KB Curator And Artifact Workflow


## Reader Outcome

After this section, the reader can:

- Пояснити, чому куратор знань корисний у пропрієтарній ERP-роботі.
- Використовувати робочий процес куратора для перетворення сирих свідчень на сторінки таблиць, контрактів, використання та патернів.
- Відокремлювати підтверджені факти, виведене значення, runtime-unverified поведінку та заблоковані свідчення.
- Підтримувати життєвий цикл артефактів від сирого до обробленого.
- Вирішувати, які інструкції куратора універсальні, а які специфічні для проєкту.

## Core Explanation

Шаблон workspace дає агенту місця для розміщення знань. Робочий процес куратора пояснює, як знання туди потрапляють.

У пропрієтарній ERP-роботі вихідні свідчення часто великі, нерівномірні та дорогі для читання: експортовані схеми таблиць, платформні бібліотеки, бібліотеки проєкту, приклади використання, документація користувача, runtime-спостереження та вихідні артефакти. Агент завдання не повинен повторно відкривати ту саму форму API чи значення таблиці. Куратор перетворює свідчення на компактні сторінки, які майбутні агенти можуть дешево шукати.

Цей проєкт використовує для цього навичку `kb-curator`. Її важливі універсальні ідеї:

- спершу шукати наявне знання;
- оглядати сирі свідчення лише тоді, коли куроване знання відсутнє;
- писати найменшу корисну сторінку;
- оновлювати індекси;
- зберігати невизначеність замість того, щоб спекулювати;
- тримати сторінки оптимізованими для пошуку агентом, а не відшліфованою прозою для людини.

Точні вихідні корені специфічні для проєкту. У цьому проєкті платформні чи проєктні бібліотеки та приклади використання мають усталені папки. Інша команда може використовувати натомість документацію API, репозиторії джерел, експорти бази даних, логи чи runtime-спостереження.

## Curator Instruction Template

Загальна інструкція куратора може виглядати так:

```markdown
# KB Curator

## Goal

Maintain an agent-first knowledge base for cheap lookup of contracts, functions, tables, usage flows, and patterns.

## Sources

- `<contract-source-root>/`: source evidence for APIs, libraries, or platform behavior.
- `<usage-source-root>/`: examples of project usage and integration conventions.
- `<artifact-root>/`: exported schemas, reports, or other source artifacts.
- `<knowledge-root>/`: durable curated knowledge produced from evidence.

## Workflow

1. Search curated knowledge first.
2. If the answer is missing, inspect the smallest relevant source evidence.
3. Create or update one focused page.
4. Link related contracts, tables, usage flows, patterns, and source evidence.
5. Update affected `INDEX.md` files.
6. Mark uncertainty explicitly.
```

Потім додайте специфічні для проєкту правила:

```markdown
## Project-Specific Rules

- Example: Use the project's parser for `<TABLE_EXPORT_FORMAT>` artifacts.
- Example: Preserve source artifact encoding; do not rewrite raw exports.
- Example: Use neutral evidence wording when source-origin details are sensitive.
- Example: When runtime testing is unavailable, record static checks and runtime-unverified assumptions.
```

## Page Families

Куратор повинен створювати чи оновлювати такі сім'ї сторінок:

- сторінки концепції модуля: бізнес-призначення, межі, терміни, шаблони наміру;
- сторінки таблиць: ключі, важливі поля, бізнес-значення, коди, пов'язані таблиці, посилання used-by;
- сторінки контрактів: публічна форма, вхідні дані, вихідні дані, побічні ефекти, runtime-припущення;
- сторінки використання: сценарій, виклики, таблиці, контракти, застереження;
- сторінки патернів: повторювані міжмодульні чи специфічні для модуля технічні форми;
- індекси: короткі навігаційні сторінки з псевдонімами та високосигнальними посиланнями.

Вони відповідають стартовим шаблонам із попереднього розділу. Якщо гайд називає сім'ю сторінок, стартовий комплект повинен включати шаблон чи пояснювати, чому ця сім'я сторінок опціональна.

## Artifact Lifecycle

Використовуйте суворий робочий процес від сирого до курованого до обробленого:

1. Розмістіть щойно знайдені експорти в зоні сирих артефактів.
2. Визначте формат і кодування перед розбором.
3. Розберіть структурованим парсером, коли можливо.
4. Перегляньте згенеровані сторінки на предмет помилок вилучення.
5. Додайте бізнес-значення, посилання та мітки невизначеності.
6. Оновіть індекси та реєстр артефактів.
7. Переміщуйте артефакти в оброблені лише після того, як пов'язані куровані сторінки створені чи оновлені.
8. Зафіксуйте результат і залишкові припущення в стані сесії.

Сирі артефакти — це свідчення. Куровані сторінки — це робоча пам'ять. Оброблені артефакти — це свідчення, яке вже розпізнане та пов'язане.

Життєвий цикл ітеративний. Оброблений артефакт може бути витіснений новішим експортом, runtime-спостереження може виправити статичний висновок, а завдання може виявити, що сторінка занадто широка чи занадто нечітка для пошуку. Куратор повинен переглядати найменші уражені сторінки, зберігати слід свідчень і фіксувати, чи старе твердження було підтверджене, звужене, виправлене чи залишене runtime-unverified. Перегляд — це не провал курації; це те, як статичне свідчення стає надійною робочою пам'яттю.

## Reproducible Procedure

1. Почніть із потреби у знаннях.
   - Назвіть завдання, модуль, таблицю, клас, сценарій чи термін користувача, що вимагає знання.
   - Уникайте нормалізації свідчень лише тому, що вони існують.

2. Шукайте дешево.
   - Спершу шукайте у wiki модуля, спільній wiki, індексах і наявних контрактах.
   - Використовуйте точні назви, псевдоніми, бізнес-терміни та ймовірні скорочення.

3. Оглядайте найменші релевантні свідчення.
   - Використовуйте вихідні корені, приклади, експорти таблиць чи runtime-нотатки згідно з правилами проєкту.
   - Надавайте перевагу структурованим парсерам для структурованих артефактів.
   - Читайте лише те, що потрібно для поточної сторінки.

4. Пишіть чи оновлюйте найменшу корисну сторінку.
   - Використовуйте відповідний шаблон.
   - Підсумовуйте поведінку та значення.
   - Посилайтеся на свідчення замість копіювання великих тіл джерела.

5. Пов'яжіть сторінку.
   - Оновіть файли `INDEX.md`.
   - Додайте псевдоніми та пов'язані посилання.
   - Пов'яжіть із концепцією модуля, контрактами таблиць, контрактами класів чи сценаріями використання, коли релевантно.

6. Зберігайте невизначеність.
   - Використовуйте `status` сторінки для повноти, наприклад `partial`, коли сторінка корисна, але не завершена.
   - Використовуйте `confidence` твердження для сили свідчень: `confirmed`, `inferred`, `runtime-unverified` чи `blocked-by-access`.
   - Позначайте runtime-unverified поведінку окремо від підтвердженої джерелом структури.
   - Зазначте перевірку, яка вирішила б невизначеність.

7. Переглядайте попереднє знання, коли свідчення змінюються.
   - Знайдіть наявні сторінки, що цитують чи залежать від артефакту, таблиці, контракту чи сценарію.
   - Оновлюйте лише найменші уражені сторінки та індекси.
   - Позначайте витіснені інтерпретації замість мовчазного видалення корисної історії.
   - Тримайте нотатку стану сесії зосередженою на змінених сторінках і залишкових припущеннях.

8. Завершіть облік артефактів.
   - Переміщуйте розпізнані сирі артефакти в оброблені лише після того, як куровані сторінки оновлені.
   - Оновіть індекс артефактів.
   - Оновіть стан сесії з тим, що змінилося і що залишається невідомим.

## Required Artifacts

- Курована сторінка wiki або явна нотатка про те, що довготривала сторінка не була потрібна.
- Оновлений індекс для будь-якої папки, що отримала чи змінила сторінку.
- Запис індексу вихідних артефактів, коли артефакти введені чи оброблені.
- Посилання між сторінками таблиць, контрактів, використання, патернів і концепції модуля.
- Нотатка стану сесії для неконверсаційної курації.
- Нотатка верифікації для перевірок парсера/статичних перевірок і невирішених runtime-припущень.
- Нотатка перегляду, коли нове свідчення змінює наявну куровану сторінку.

## Example

Проєкт отримує експорт таблиці для бюджетної ієрархії, а завдання запитує про доступ до батьківських бюджетних підрозділів.

Куратор не повинен вставляти експорт у сторінку wiki. Він повинен:

- розмістити експорт у папці сирих артефактів;
- визначити кодування та формат;
- розібрати його структурованим парсером, якщо такий існує;
- створити чи оновити контракт таблиці для таблиці ієрархії;
- додати псевдоніми, такі як бюджетний підрозділ, батьківський підрозділ та ієрархія;
- пов'язати таблицю з контрактом допоміжного методу доступу та сценарієм використання доступу;
- позначити бізнес-значення як partial, якщо його підтверджують лише назви полів;
- перемістити артефакт в оброблені лише після того, як сторінка таблиці та індекс оновлені;
- зафіксувати залишкові runtime-припущення, такі як реальне налаштування прав доступу чи поведінку активних дат.

Це створює короткий маршрут для майбутніх агентів: термін користувача -> концепція модуля -> сценарій використання -> сторінки таблиці та контракту -> вихідний артефакт.

## Failure Modes

- Читання сирих свідчень перед перевіркою наявного курованого знання.
- Генерація сторінок без посилань чи індексів.
- Розгляд вилучених назв полів як підтвердженого бізнес-значення.
- Переміщення артефактів в оброблені до того, як куровані сторінки переглянуто.
- Додавання нових свідчень без перевірки, чи потребують перегляду старі куровані сторінки.
- Копіювання великих тіл джерела в сторінки wiki.
- Створення відшліфованої прози, занадто довгої для прийому завдання.
- Витрачання токенів на спекуляцію замість позначення невизначеності.
- Дозвіл специфічним для проєкту парсерам чи вихідним кореням виглядати універсальними в опублікованому гайді.

## Assumptions And Runtime Limits

- Куратор може зробити статичне свідчення дешевим у використанні, але не може довести кожну runtime-поведінку.
- Деякі свідчення можуть бути заблоковані доступом, ліцензією, середовищем чи доступністю runtime.
- Згенеровані сторінки вимагають перегляду; вивід парсера не є автоматично довготривалим знанням.
- Інший проєкт може замінити `kb-curator` цього проєкту скриптами, людським переглядом чи іншим інструментом, якщо це зберігає ту саму дисципліну свідчень.
