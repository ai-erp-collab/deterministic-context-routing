# Public Article Outline

## Working Title

Як зробити AI-агентів корисними у великих програмних проєктах, насичених доменною логікою

## Thesis

AI-агенти стають надійнішими, коли проєкт дає їм довготривалу файлову систему знань, яка спрямовує їх до необхідного та достатнього контексту для кожного завдання.

## Outline

1. Реальний блокер — не лише генерація коду.
   - Великі системи ховають поведінку в конвенціях, схемах, runtime-припущеннях, бізнес-мові та історичних рішеннях.
   - Дати агенту все — занадто дорого і часто менш корисно, ніж дати йому правильний невеликий контекст.

2. Розглядайте документацію як маршрутизацію контексту.
   - Концепції модуля пояснюють призначення та межі.
   - Сторінки контрактів описують таблиці, API та поведінку.
   - Сценарії використання пов'язують об'єкти в бізнес-сценарії.
   - Стан сесії тримає роботу придатною для відновлення.
   - Git, компілятори, тести та runtime-перевірки посилюють верифікацію, коли доступні, але не замінюють пам'ять безперервності завдання.
   - Мітки впевненості повідомляють наступному агенту, які твердження підтверджені, виведені, runtime-unverified чи заблоковані доступом.
   - Документація — це інфраструктура маршрутизації для майбутніх агентів, а не прибирання після реалізації.

3. Тримайте свідчення окремо від активного контексту.
   - Сирий матеріал може бути великим.
   - Контекст активного завдання повинен бути малим.
   - Куровані сторінки роблять повторювану роботу дешевшою.

4. Впроваджуйте через один реальний модуль.
   - Оберіть один модуль.
   - Куруйте один артефакт.
   - Виконайте одне завдання за підтримки робочого процесу.
   - Зафіксуйте верифікацію та припущення.
   - Розширюйте після того, як метод доведе себе.

5. Цикл якості.
   - Кожне завдання повинно робити workspace легшим для розуміння.
   - Нотатки верифікації важливі, коли runtime-перевірки обмежені.
   - Відомі проблеми запобігають повторюваним помилкам.

## Public-Safe Language Rules

- Кажіть "платформні бібліотеки", "бібліотеки проєкту", "приклади джерела", "runtime-спостереження", "експортовані схеми таблиць", "документація користувача" та "бізнес-вимоги".
- Не розкривайте чутливі деталі походження.
- Тримайте приклади загальними, якщо тільки конкретний приклад уже не затверджений для публічного використання.

## Possible Ending

Мета — не побудувати ідеальну енциклопедію. Мета — дати агенту достатньо контексту, щоб безпечно міркувати сьогодні, і залишити після себе достатньо довготривалого знання, щоб завтрашнє завдання починалося ближче до істини.
